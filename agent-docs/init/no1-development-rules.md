# 開発ルール

## 前提条件
- 開発するアプリは必ず `app.py` 1 ファイルで完結させる。複数ファイルへの分割は禁止
- Gradio 6 の新 API を使う（旧 API は使用禁止）
- メインのUIは gr.HTML で構築する。シンプルなフォームやボタンは必要に応じて Gradio のコンポーネントを使う

---

## 開発ワークフロー
### 0. 初期化
- 新しいセッションを開始したら、最初にポート 7860 が空いていることを確認する。
  - macOS / Linux: `lsof -i :7860`
  - Windows: `netstat -ano | findstr :7860`
- もしプロセスが存在する場合は強制終了する。
  - macOS / Linux: `kill -9 <PID>`
  - Windows: `taskkill /F /PID <PID>`

### 1. 機能実装 → アプリ起動サイクル
- ユーザーから受けた機能依頼を `app.py` を実装する
- 実装後、アプリが起動していなければ `uv run gradio app.py` でアプリを起動
- まずは自分で動作確認をし、問題なければユーザーに動作確認してもらう
- `gradio` コマンドはファイル変更を検知して自動リロードするため、以降の変更は即座に反映される
- サイクル: **ユーザーの指示 → `app.py` を修正 → 自動リロード → ユーザーが確認**

### 2. プランニングファースト
- 3 ステップ以上のタスクや設計判断が必要な場合は必ずプランモードに入る
- 問題が発生したら即停止・再プラン — 押し進めない

### 3. サブエージェント戦略
- メインコンテキストを汚さないためサブエージェントを積極活用する
- リサーチ・探索・並列分析はサブエージェントに委ねる
- 1 サブエージェント = 1 タスク に徹する

### 4. 完了前の検証
- 動作確認なしにタスク完了とマークしない
- 「スタッフエンジニアが承認するか？」と自問する
- アプリが起動済みであることを確認する

### 5. エレガントさを要求（バランス重視）
- 非自明な変更では「より優雅な方法はないか？」と一時停止する
- ハック的な修正に気づいたら: 「今知っていることすべてを踏まえてエレガントな解を実装する」
- 明白な小さな修正には過剰設計しない

### 6. 自律的バグ修正
- バグレポートを受けたら即修正。ハンドホールドを求めない

---

## 環境変数の管理
- `.env` はエージェントが管理する。ユーザーは手動編集しない
- 外部サービスのAPIキーや、データベースのパスワード管理等が必要になった場合は、必要に応じてユーザーに値を確認し`.env`を修正すること。
- ただし、ユーザーの同意なしに`.env`内の既存キーを勝手に削除・変更しないこと。

---

## 依存パッケージの管理
- 新しいライブラリが必要になったら `pyproject.toml` の `dependencies` に追記し、`uv sync` を実行する

---

## コードスタイル規約

### Python
- **型ヒント**: 関数シグネチャには型ヒントを付ける
- **定数**: 大文字スネークケース（例: `MAX_TOKENS = 4096`）
- **環境変数**: `python-dotenv` で `.env` から読み込む
- **コメント**: ロジックが自明でない箇所のみ（docstring 不要）
- **インポート順**: 標準ライブラリ → サードパーティ → ローカル

### Gradio
- コンポーネントには `label=` を必ず付ける
- `gr.Blocks()` の中は UI 定義 → イベントハンドラ登録の順で書く

### 禁止
- 不要な抽象化・ユーティリティクラスを作らない
- 使われていないコードを残さない
- `print` デバッグを残したままにしない

## コアプリンシプル

- **手抜きなし**: 根本原因を探す。一時的な修正をしない。シニアエンジニアの基準で
- **最小インパクト**: 必要な箇所だけ変更する。バグを混入させない
